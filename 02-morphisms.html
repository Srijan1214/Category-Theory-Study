Why This Matters for Functional Programming

Your functions have "types" of properties. Learning them helps you write safer code.

  • Isomorphisms: Two types that are "really the same." Example: `Name` as a newtype wrapper around `String` is isomorphic to `String`, but the type system treats them differently—this is type safety! You can't accidentally mix up names and passwords.
  • Monomorphisms (injective): These functions lose NO information. If you know the output, you know exactly what the input was. This is why parsing is powerful: a parser is a mono from raw strings to your data structure. You can always verify it worked correctly.
  • Epimorphisms (surjective): These functions cover all possibilities in the target. Pattern matching on an `Option` value uses epis—you might have a function that applies to "all possible Option outcomes." When you write exhaustive case branches, you're handling all epimorphic targets.
  • Endomorphisms: Functions from a type to itself. These compose nicely (monoid structure). Folding a list is an endomorphism composition. Recursive functions are endomorphisms. The fact that they form a monoid means you can combine them algebraically.

Initial and Terminal Objects are the key to handling edge cases:

  • Terminal object (Like `Unit` or `()`): The "void return"—the type with no information. Having exactly one function to it means every function's return can be "collapsed" to nothing. This models functions that only care about side effects.
  • Initial object (Like `Nothing`, `None`, or `undefined`): The empty type. Functions FROM it to anywhere can handle "no value." In Haskell, this is part of how `Maybe` and `Either` work—they give you a way to indicate failure or absence.

The payoff: By understanding these morphism types, you learn to recognize patterns. "Is this function injective? Can I safely invert it?" becomes a question you can answer by looking at types, not running code.

Special Types of Morphisms

Not all morphisms are equal. Some have special properties that tell us about the structure they preserve. Let's look at the main types.

Isomorphism (Two-Way Invertible)

An isomorphism f : A → B is a morphism that has an inverse g : B → A such that:

  • g ∘ f = id_A (going backwards then forwards gives you back what you started with)
  • f ∘ g = id_B (going forwards then backwards gives you back what you started with)

If such an f exists, we say A and B are isomorphic, written A ≅ B. Isomorphic objects are "the same" from a categorical perspective—they have identical structure.

In Set: Isomorphisms are bijections (one-to-one AND onto functions). Two sets are isomorphic if they have the same cardinality (size).

In Vect_k: Isomorphisms are invertible linear maps. Two vector spaces are isomorphic if they have the same dimension.

In Functional Programming: Isomorphisms let you safely switch between representations. If you have an isomorphism between `String` and `List Char`, you can freely convert between them without losing data. This is why newtypes (wrapper types) are so useful in Haskell:

newtype UserId = UserId Int

-- UserId ≅ Int (isomorphic)
-- But type system keeps them separate for safety
-- You can't accidentally pass a UserId where an Int is expected

Monomorphism (Injective / One-to-One)

A morphism m : A → B is a monomorphism if it is left-cancellative. This means:

For any morphisms f₁, f₂ : X → A, if m ∘ f₁ = m ∘ f₂ then f₁ = f₂.

Intuition: You can "cancel" m from the left. If two things give the same result after applying m, they must have been the same to begin with. This is like saying m is one-to-one (injective): different inputs give different outputs.

In Set: Monos are injective (one-to-one) functions. For example, f(x) = 2x is a mono from ℕ to ℕ (different natural numbers map to different even numbers), but f(x) = x mod 2 is not (both 0 and 2 map to 0).

In Functional Programming: Monomorphisms are your "information-preserving" functions. Parsers are monos: if a parser successfully turns a string into a value, that value uniquely determines what the string was (ignoring whitespace, formatting, etc.). This is why you can reliably reverse them:

parseInt : String → Option Int -- mono (if it parses successfully, input was uniquely an integer)

-- If you know the output was 42, you know the input was some valid "42" encoding
-- This property is why parsing/validation is safe!

Epimorphism (Surjective / Onto)

A morphism e : A → B is an epimorphism if it is right-cancellative. This means:

For any morphisms f₁, f₂ : B → X, if f₁ ∘ e = f₂ ∘ e then f₁ = f₂.

Intuition: You can "cancel" e from the right. If applying e, then something else, gives two different results only if those two things were different. This is like saying e is onto (surjective): every element in B is "hit" by some element in A.

In Set: Epis are surjective (onto) functions. For example, f : ℝ → [0, ∞) where f(x) = x² is surjective (every non-negative number is a square of something), but f : ℕ → ℕ where f(x) = 2x is not (no odd numbers are hit).

In Functional Programming: Epimorphisms are your exhaustive handlers. Pattern matching on sum types uses epis. When you write:

handle : Option Int → String
handle Nothing = "no value"
handle (Just x) = "got " ++ show x

-- This is an epi: it "covers" all possible Option values
-- The compiler forces you to handle both cases
-- Missing a case is a compile error!

Endomorphism (Arrow to Self)

An endomorphism is simply a morphism f : A → A from an object to itself.

Key insight: The set of all endomorphisms on an object forms a monoid (structure with a binary operation and identity). You compose them: if f : A → A and g : A → A, then g ∘ f : A → A is also an endomorphism. And id_A : A → A is always an endomorphism.

Identity Morphism vs. Other Endomorphisms

Remember from Chapter 1: every object must have an identity morphism. But an endomorphism is ANY morphism from an object to itself. So:

  • Identity morphism `id_A`: Does nothing: `f ∘ id_A = f` for all f. There's exactly one per object.
  • Other endomorphisms: There can be many. They have different composition behaviors.

Concrete example in the Type category (from Chapter 1): Let's look at the type `Int`:

-- Identity endomorphism on Int:
id : Int → Int = λx. x

-- Other endomorphisms on Int (many of them!):
double : Int → Int = λx. 2x
addOne : Int → Int = λx. x + 1

-- Importantly: they compose DIFFERENTLY when order changes!
(double ∘ addOne) 5 = double(addOne(5)) = double(6) = 12
(addOne ∘ double) 5 = addOne(double(5)) = addOne(10) = 11

-- These are NOT the same! Order of composition matters.
-- In category theory, this is why different endomorphisms are distinguished:
-- by their composition behavior, not by their "internal effect"

The abstract insight: In category theory, we don't care what's "inside" Int. We only know Int through its morphisms. Different endomorphisms on Int are different precisely because they compose with other morphisms in different ways. That's how the abstract category distinguishes them.

Why this matters: Endomorphisms are important because monads are defined using endofunctors, which are endomorphisms at the level of categories.

In Functional Programming: Endomorphisms are transformations on a value. They're your building blocks for loops, recursion, and folds. The fact that they form a monoid means:

double : Int → Int -- endomorphism on Int
addOne : Int → Int -- endomorphism on Int

-- Compose them:
addOneThenDouble = double ∘ addOne : Int → Int -- also an endomorphism

-- This monoid structure is why fold works:
-- Folding a list is just composing endomorphisms!
-- fold: (A → A) → [A] → A (simplification)

Initial and Terminal Objects

These are special "universal" objects. They have a unique property: there is exactly one morphism between them and any other object. This uniqueness is what makes them powerful.

Terminal Object (The "End Point")

An object 1 (or *) in a category C is a terminal object if:

  • For every object A in the category, there exists exactly one morphism A → 1.

Intuition: Think of 1 as a "gathering point"—everything eventually points to it, and there's only one way to do so. It's like saying "no matter what object you start with, there's a unique path to 1."

In Set: A terminal object is any singleton set like {*} or {42}. Why? Because for any set A, there's exactly one function from A to {*}: the function that maps every element of A to the single element *. You can't define two different functions from A to a singleton—there's only one choice.

Example: f : {1, 2, 3} → {*} must send all three elements to *, so f(1) = *, f(2) = *, f(3) = *. There's no other function.

Initial Object (The "Starting Point")

An object 0 (or ∅) in a category C is an initial object if:

  • For every object A in the category, there exists exactly one morphism 0 → A.

Intuition: Think of 0 as a "starting point"—you can begin from it and go uniquely to any other object. It's the dual of terminal.

In Set: The initial object is the empty set ∅. Why? Because there is exactly one function from ∅ to any set A: the empty function (which does nothing, sends no elements). You can't define two different empty functions—there's only one.

Why? A function is defined by saying where it sends each element. The empty set has no elements, so any function from ∅ is completely defined by doing nothing.

What to Be Careful About

Beyond just "arrows coming out/in," here are key things to remember:

  • Uniqueness is absolute: Not just "there exists one arrow," but "there exists exactly one arrow." If you can find two different morphisms A → 1, then 1 is not terminal. Uniqueness is non-negotiable.
  • Terminal/Initial may not exist: Not every category has a terminal or initial object. For example, the category of groups has an initial object (the trivial group), but some categories don't.
  • Terminal/Initial are unique up to isomorphism: If two objects both satisfy the terminal property, they must be isomorphic. So "the" terminal object is unique, though there may be multiple isomorphic copies of it.
  • They're defined by their universal property: Initial and terminal objects are characterized by having arrows to/from all other objects. This is more important than their "internal structure." Two very different-looking objects can both be terminal objects as long as they have this property.
  • Context matters: What is terminal in Set is very different from what is terminal in Vect_k. In Vect_k, the terminal object is the one-dimensional vector space {0} (the zero vector space).

Why This Matters

Morphisms with special properties (isos, monos, epis) tell us about the structure of a category. Initial and terminal objects are "universal" structures that appear everywhere in mathematics. Commutative diagrams become the universal language for expressing equalities in category theory—they're easier to reason about than nested function notation.
