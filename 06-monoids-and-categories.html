Why This Matters for Functional Programming

Understanding monoids is the key to understanding how values combine and fold. This is critical for practical FP.

Monoids model composable operations: Any time you can combine two values into one (addition, string concatenation, list merging), you have a monoid structure. Functional languages exploit this heavily.

Example: Summing a list is monoid action (0 is the identity, + is associative). Folding a tree is monoid action. Reducing/aggregating in MapReduce is monoid action. All of these are instances of the same abstract pattern!

Why endomorphism monoids matter for monads:

  • Every monad is secretly a monoid in the category of endofunctors
  • Understanding monoids helps you understand how monads compose
  • The unit/multiplication of a monad follow monoid laws
  • This explains why monad laws are what they are

Practical consequence: When you understand that `>>=` (bind) is just monoid multiplication for effects, you understand why monads work the way they do. Effects compose just like values in a monoid!

What is a Monoid?

A monoid (M, ⋆, e) is a set M equipped with:

  • A binary operation ⋆ : M × M → M (called composition or multiplication).
  • An identity element e ∈ M.

These must satisfy:

  • Associativity: (a ⋆ b) ⋆ c = a ⋆ (b ⋆ c) for all a, b, c ∈ M.
  • Identity: e ⋆ a = a = a ⋆ e for all a ∈ M.

Monoid Examples

Integers under addition (ℤ, +, 0): Any integer can be added to any other, and 0 is the identity.
Strings under concatenation (Σ*, ++, ""): Strings form a monoid where ++ is concatenation and "" is the empty string.
Functions under composition: All functions X → X form a monoid under composition, with the identity function as the unit.

Monoid as a Category

Here's the key insight: any monoid can be viewed as a category with a single object.

Given a monoid (M, ⋆, e), construct a category C with:

  • One object: Call it •.
  • Morphisms: Hom(•, •) = M (the elements of M are morphisms from • to itself).
  • Composition: For morphisms m₁, m₂ ∈ M, their composition is m₂ ∘ m₁ = m₂ ⋆ m₁.
  • Identity: The identity morphism id_• = e.

The monoid laws (associativity and identity) ensure this forms a valid category.

Endomorphisms as a Monoid

Conversely, if we have a category C with an object A, the set of all endomorphisms End(A) = Hom(A, A) (any morphism from an object to itself) forms a monoid under composition:

  • Operation: composition of morphisms.
  • Unit: the identity morphism id_A.

This is always a monoid because:

  • Composition is associative (by the category axioms).
  • id_A is a left and right identity (by the category axioms).

Example: Endomorphisms of a Vector Space

Linear endomorphisms: Let V be a vector space. The set of linear maps V → V forms a monoid under composition. This is actually a ring (since we also have pointwise addition), but forgetting addition, it's a monoid. Matrix multiplication is a concrete example when V = ℝⁿ.

Why This Matters for Monads

The deep categorical insight is this: just as a monoid is a category with one object, a monad is a monoid in the category of endofunctors.

Specifically:

  • The "category" is now the category of endofunctors on C (denoted [C, C]).
  • The "objects" are endofunctors T : C → C.
  • The "morphisms" are natural transformations between endofunctors.
  • The "product" is functor composition.
  • A "monoid" in this category is a triple (T, η, μ) where:
    • T is the "object" (an endofunctor).
    • η : Id ⇒ T is the "identity morphism" (unit natural transformation).
    • μ : T ∘ T ⇒ T is the "multiplication" (multiplication natural transformation).

This perspective unifies monoids and monads: both are monoids in some category, just at different levels of abstraction.
