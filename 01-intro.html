01 — What is a Category?

Home Next →

TL;DR: Category theory is the math of composition. Your code is already categorical—you just might not have realized it!

Start Here: You Already Know This

Here's the intuition: A category is a way to talk about stuff and how stuff relates to other stuff.

In your code, you already have categories everywhere:

  • Objects = Types (Int, String, User, Maybe Bool, List String...)
  • Arrows = Functions (anything that goes from one type to another)
  • Composition = When you call functions and pass the result to another function

A category just formalizes this. And the reason we formalize it? Because once you see the pattern, you unlock a superpower: you can reason about your code mathematically.

What You Need to Know About Categories

A category has three ingredients:

  • Objects: Things we care about (types in your language)
  • Arrows (Morphisms): Relationships between objects (functions from one type to another)
  • A way to chain arrows together: If you have an arrow from A to B, and another from B to C, you should be able to chain them to get from A to C

The Two Rules (That Aren't Really Rules, Just Common Sense)

For everything to work, we need just two properties:

  1. Identity: Every type/object should have a "do nothing" function. In most languages, it's literally x => x.
  2. Associativity: When you chain three functions, it shouldn't matter which two you chain first. (h ∘ g) ∘ f should give the same result as h ∘ (g ∘ f).

Think about it: these are just saying "composition works the way it should." They're not surprise requirements—they're exactly how functions work in real code!

Examples: Different Flavors of Categories

Example 1: Your Programming Language (The Type Category)

This is the one that matters most for you. Every time you write a function, you're working in a category.

Objects: Types in your language: `Int`, `String`, `Bool`, `User`, `List Int`, `Maybe String`, etc.

Arrows (Morphisms): Functions. A function `f : Int → String` is an arrow from the Int type to the String type. Pure functions always preserve this structure.

Composition: When you chain functions: `getName(parseUser(jsonString))`. That's categorical composition. You composed two functions to make a bigger one.

Identity: The identity function: `id x = x`. It does nothing—it's there by default.

Why this matters: You're literally building programs as paths through the type category. The fact that you can refactor `(h ∘ g) ∘ f` to `h ∘ (g ∘ f)` without changing behavior? That's the associativity law. It's not magic—it's just how functions work.

-- Example in pseudocode:
parseJSON : String → Maybe User
getName : User → String
exclaim : String → String

-- Compose them naturally:
shoutName = exclaim ∘ getName ∘ parseJSON
-- Type signature: String → Maybe String
-- This composition is mathematically guaranteed to work
-- because we're in the type category!

🎓 Learning Strategy: Because you're an experienced programmer, we'll use the Type Category as your primary lens throughout this guide. You can verify intuitions by writing and running code. However, to build true categorical intuition, we'll also occasionally step back to explore simpler categories like Poset (Chapter 1) where the abstract scaffolding becomes visible. This teaches you that categorical patterns work in *many* different settings, not just types-and-functions. The goal: learn in familiar territory, but glimpse the universal principles.

Example 2: Set (The Foundation of Everything)

Before computers, mathematicians had categories. The first one was **Set**: just sets and functions.

Objects: Any set you can imagine: {red, green, blue}, {1, 2, 3, ...}, all real numbers ℝ, the power set of your favorite set, etc.

Arrows: Functions between sets. If you have a function `f : A → B`, then for every element in A, there's exactly one element in B it maps to.

Composition & Identity: Same as before—function composition, identity function.

Why this is foundational: Your type category is actually a special case of Set! The types in your language are sets. Functions between types are functions between sets. If you understand Set, you understand the mathematical foundation of all programming.

Example 3: Vect_k (Structure That Matters: Vector Spaces)

Sometimes you want more structure. Here's what happens when you care about that structure.

Objects: Vector spaces (not just arbitrary sets, but sets with linear structure). Like ℝ², ℝ³, the space of all polynomials, etc.

Arrows: **Linear maps** (functions that respect the structure). A function `f : V → W` is linear if:
  • `f(v₁ + v₂) = f(v₁) + f(v₂)` — addition is preserved
  • `f(c·v) = c·f(v)` — scaling is preserved
It's not *any* function—only functions that play nice with the vector structure.

The key insight: When you move to Vect_k, you **restrict** which functions are allowed. Only functions that preserve linear structure count as morphisms. This is more restrictive than Set, but more powerful because you know your functions respect the structure you care about. This is a recurring theme: stronger categories capture more specific problems.

Example 4: Poset (When Things Have an Order)

Objects: Elements in a partially ordered set. Like the numbers {1, 2, 3, 4, 5} with their usual ≤ ordering, or a hierarchy of types in your language.

Arrows: There's at most *one* arrow `a → b` if `a ≤ b`, and no arrow otherwise. This might sound weird, but it captures ordering perfectly.

Composition & Identity: If `a ≤ b` and `b ≤ c`, then `a ≤ c` (transitivity). The identity is reflexivity: `a ≤ a`.

Why this design trick works: We don't need "real" functions here. We just need to capture the relationship "less than or equal to." Using arrows represents that relationship. It's minimal but captures exactly what we need.

Example 5: Cat (Meta—Categories as Objects)

Want to blow your mind? Categories themselves can be objects in a category.

Objects: Categories like Set, Vect_k, Poset, or any other category you've defined.

Arrows: Functors (you'll learn these in Chapter 4). A functor is to categories what a function is to sets—it maps objects to objects and morphisms to morphisms while preserving structure.

Composition & Identity: Functor composition, identity functor.

Why this is wild: This is self-reference at its finest. We're using category theory to study category theory. It shows the power of the abstraction—the same patterns work at every level.

Poset (Partially Ordered Sets as Categories)

What it is: A poset is a set with a partial ordering (≤). We can view any poset as a category!

What's a partial order? A relation ≤ where:
  • Reflexive: x ≤ x (everything relates to itself)
  • Antisymmetric: if x ≤ y and y ≤ x, then x = y
  • Transitive: if x ≤ y and y ≤ z, then x ≤ z
Examples: (ℕ, ≤), (ℝ, ≤), the divisibility relation on integers, the subset relation ⊆.

Objects: Elements of the poset (e.g., if poset is {1, 2, 3} with ≤, the objects are 1, 2, 3)
Morphisms: For each pair x, y, there is exactly one arrow x → y if and only if x ≤ y. If x ≰ y, there is no arrow.
Composition: If there's an arrow x → y and an arrow y → z, then transitivity guarantees there's an arrow x → z, and composition is automatic.
Identity: The reflexive property ensures id_x exists (since x ≤ x).

Concrete example: The poset {1, 2, 3} with ≤:
  • Arrow 1 → 2 (because 1 ≤ 2)
  • Arrow 2 → 3 (because 2 ≤ 3)
  • Arrow 1 → 3 (because 1 ≤ 3) — this is the composition of 1 → 2 → 3
  • No arrow 3 → 2 (because 3 ≰ 2)
Why this matters: Posets are very "thin" categories (at most one arrow between any two objects). They show that categories are very general—even abstract relations can be viewed as categories.

Key Intuition

A category is an abstract notion of "things" (objects) and "structure-preserving relationships" (morphisms) between them. The laws ensure that relationships can be chained coherently.

When Bartosz Milewski starts with "we have objects and arrows," he's describing exactly this setup. The associativity and identity laws are what make it a category.

Why This Matters for Functional Programming

Programming IS category theory in disguise. This is the big insight:

Types form a category: In functional programming, your types (Int, String, Bool, List, etc.) are the objects, and your functions are the morphisms. When you write a function signature f : Int → String, you're writing a categorical arrow. When you compose two functions, you're using categorical composition!

Example: getUserName : User → String and shout : String → String compose to shout(getUserName(user)) : User → String. This is exactly categorical composition in your code.

Why this understanding changes everything:

  • Composition is your primary tool. In functional programming, you don't solve problems with loops and mutable state—you solve them by composing small, pure functions. Category theory formally proves why this works and how to do it correctly.
  • Types guarantee correctness. A function's type signature tells you what it can and cannot do. Category theory explains why types are so powerful: they're not just annotations—they're mathematical structure that your compiler can verify.
  • Effects have structure too. Pure functions live in one "world" (the category Set). But what about side effects like I/O, error handling, or state? Category theory says: these form *different* categories! Functors let you systematically translate between worlds. This is the entire foundation of monads.
  • Equational reasoning actually works. Because categories enforce composition and associativity, you can reason about your programs algebraically, not just by running them. "Will this function composition work?" becomes provable, not guesswork.
